# Cline

## О сервисе

Cline — это open source AI coding agent, который работает прямо в редакторе или в терминале. По официальной документации Cline умеет читать и изменять файлы, запускать команды, использовать браузер, подключать внешние инструменты через MCP servers и вести разработку в формате обычного диалога.

Для BYOK-сценария это особенно удобно, потому что Cline не привязывает вас к одному закрытому провайдеру. Вы можете выбрать собственный API provider, указать свой endpoint, вставить свой API key и работать с той моделью, которая лучше подходит именно под ваш coding workflow.

Ниже страница сфокусирована на варианте, где Cline подключается к `https://leantoken.tech` через `OpenAI Compatible` provider. Именно этот путь лучше всего совпадает с текущим экраном настроек Cline и с официальной документацией по OpenAI-compatible конфигурации.

## Когда Cline особенно полезен

Cline хорошо подходит для задач, где нужен не просто чат с моделью, а полноценный агентный workflow внутри IDE.

- разбор незнакомого репозитория с чтением файлов, поиском по коду и объяснением архитектуры
- реализация новых фич, когда нужно сначала обсудить план, а потом перейти к изменению файлов
- отладка багов через связку терминала, логов, тестов и пошаговых правок
- рефакторинг, где полезно быстро сравнивать подходы и откатываться через checkpoints
- работа с веб-приложениями, когда нужно проверять интерфейс через browser tool
- интеграции с внешними системами через MCP servers: GitHub, документация, базы данных, таск-трекеры и внутренние инструменты команды

## Как устроена работа в Cline

Официальные материалы Cline описывают несколько базовых механизмов, которые важно понимать до настройки модели.

### Tasks

Вся работа в Cline происходит внутри задач. Одна задача хранит:

- исходный prompt
- всю переписку по задаче
- изменения файлов
- выполненные команды
- решения и промежуточный контекст
- usage, стоимость и длительность выполнения

Это удобно для длинных инженерных задач: вы можете вернуться к предыдущей работе, продолжить ее, проверить, какие файлы менялись, и не начинать каждый раз с пустого окна.

### Plan и Act

Один из самых полезных режимов Cline — разделение на `Plan` и `Act`.

- `Plan` нужен, чтобы исследовать кодовую базу, обсудить стратегию, найти затронутые файлы и продумать реализацию без изменения файлов и без запуска команд
- `Act` используется после этого, когда план уже понятен и можно переходить к правкам, командам, тестам и реализации

Практически это дает более предсказуемый результат. Для средних и больших задач лучше сначала пройти короткий planning-этап, а потом переключиться в `Act`.

### Checkpoints

Cline по умолчанию создает checkpoints. Это отдельные снимки состояния проекта, которые не загрязняют вашу обычную Git-историю. Если агент пошел не туда, вы можете восстановить файлы, саму задачу или и то и другое.

Для BYOK-сценария это важно по двум причинам:

- можно смелее разрешать Cline более автономную работу на хорошо определенных задачах
- можно быстрее тестировать разные подходы, не боясь потерять контроль над кодом

### Дополнительные возможности, которые полезно знать заранее

Помимо базовой настройки провайдера, в Cline есть несколько систем, которые сильно улучшают рабочий процесс:

- `Rules` для постоянных project-level инструкций и ограничений
- `Skills` для специализированных знаний и процедур
- `Workflows` для повторяемых многошаговых задач
- `Hooks` для автоматических проверок и защитных правил
- `.clineignore` для исключения мусорных файлов и директорий из контекста
- `Memory Bank` для структурированного хранения контекста между сессиями
- `MCP servers` для расширения Cline внешними tools и resources

Если говорить проще, Cline особенно раскрывается не на одноразовом prompt, а когда вы постепенно подстраиваете его под свою кодовую базу и свой процесс.

## Что понадобится

Для настройки Cline через BYOK обычно нужны:

- VS Code, Cursor или другой совместимый редактор, где установлен Cline
- аккаунт на `https://leantoken.tech`
- API key LeanToken
- OpenAI-compatible endpoint `https://api.leantoken.tech/v1`
- модель, которая хорошо справляется с coding-задачами и корректно работает с tool-calling workflow
- репозиторий или рабочая папка, которую вы откроете в редакторе

Ниже шаги написаны в первую очередь под сценарий VS Code/Cursor-style интерфейса, потому что именно такой экран показан на вашем скриншоте.

## BYOK на leantoken.tech

В официальной документации Cline BYOK — это путь, где вы используете собственные ключи и собственный provider вместо встроенного `Cline Provider`.

Для `leantoken.tech` это означает следующее:

1. вы получаете собственный LeanToken API key
2. выбираете в Cline тип провайдера `OpenAI Compatible`
3. указываете `https://api.leantoken.tech/v1` как `Base URL`
4. вставляете свой LeanToken key
5. выбираете model ID, который доступен через ваш LeanToken endpoint

Такой поток особенно полезен, когда вы хотите:

- сами контролировать биллинг и ключи
- использовать один endpoint для нескольких AI-инструментов, а не только для Cline
- переключаться между моделями без смены общей схемы интеграции
- держать границу приватности и доступа к моделям отдельно от встроенной подписки конкретного IDE-провайдера

Официальные материалы Cline отдельно подчеркивают еще один важный момент: API keys хранятся в системном credential manager и отправляются только выбранному провайдеру. Для LeanToken это хороший сценарий, потому что вы не строите лишнюю промежуточную схему поверх локального editor workflow.

## Как получить API key на leantoken.tech

Перед настройкой Cline получите key и endpoint в LeanToken.

![Публичная страница LeanToken с Sign up, Log in и Dashboard](../../images/leantoken/leantoken.tech.1.landing.png)

1. Откройте `https://leantoken.tech`.
2. Нажмите `Sign up`, если у вас еще нет аккаунта, `Log in`, если аккаунт уже есть, или `Dashboard`, если вы уже вошли в систему.

![Панель LeanToken с API endpoint и быстрыми ссылками](../../images/leantoken/leantoken.tech.2.dashboard.png)

3. После входа откройте dashboard или обзорную страницу API.
4. Скопируйте API endpoint. На скриншотах показан `https://api.leantoken.tech/v1`.
5. Перейдите в раздел управления ключами через `Manage keys` или `API Keys`.

![Страница API Keys в LeanToken с действием Create key](../../images/leantoken/leantoken.tech.3.key-managment.png)

6. Нажмите `Create key`.
7. Скопируйте созданный API key.
8. Сохраните и endpoint, и key в секрете. Если ключ попал в лог, скриншот, git или переписку, перевыпустите его и замените в Cline.

## Как установить Cline

По текущей официальной документации Cline поддерживает несколько платформ, включая VS Code, Cursor, JetBrains IDEs, CLI, Zed, Neovim, VSCodium и Windsurf. Для вашего скриншота и для самого прямого сценария LeanToken лучше ориентироваться на расширение в VS Code-совместимом редакторе.

Базовый путь установки такой:

1. Откройте редактор.
2. Перейдите в `Extensions`.
3. Найдите `Cline`.
4. Установите расширение.
5. Откройте Cline через иконку в боковой панели или через command palette.

Если после установки иконка не появилась:

- подтвердите разрешение на запуск расширения, если редактор его запросил
- перезапустите редактор
- откройте Cline через command palette

## Как настроить LeanToken в Cline

Ниже показан именно тот экран Cline Settings, который нужен для подключения LeanToken как OpenAI-compatible провайдера.

![Экран Settings в Cline с провайдером OpenAI Compatible, LeanToken Base URL, полем OpenAI Compatible API Key и моделью qwen3-235b](../../images/cline/cline-openai-compatible-settings.png)

На скриншоте уже видны ключевые поля:

- `API Provider`
- `Base URL`
- `OpenAI Compatible API Key`
- `Model ID`
- `Custom Headers`
- блоки `MODEL CONFIGURATION` и `ADVANCED`

### Пошаговая настройка

1. Откройте Cline в боковой панели редактора.
2. Нажмите на значок шестеренки в правом верхнем углу панели Cline.
3. Откройте раздел `API Configuration`.
4. В поле `API Provider` выберите `OpenAI Compatible`.
5. В поле `Base URL` укажите `https://api.leantoken.tech/v1`.
6. В поле `OpenAI Compatible API Key` вставьте ваш LeanToken API key.
7. В поле `Model ID` укажите нужный model ID. На вашем скриншоте в качестве примера выбран `qwen3-235b`.
8. Поле `Custom Headers` оставьте пустым, если ваша схема подключения LeanToken не требует дополнительных заголовков.
9. Переключатели `Set Azure API version` и `Use Azure Identity Authentication` не включайте, если вы используете обычный OpenAI-compatible endpoint LeanToken.
10. Нажмите `Done` или просто закройте настройки, если интерфейс уже сохранил состояние.
11. Вернитесь в чат Cline и отправьте простое сообщение для проверки соединения.

Минимальная рабочая конфигурация выглядит так:

```text
API Provider: OpenAI Compatible
Base URL: https://api.leantoken.tech/v1
OpenAI Compatible API Key: <ваш_LeanToken_API_key>
Model ID: <ваш_model_id>
```

### Что означают дополнительные поля на экране

Официальная документация Cline по OpenAI-compatible провайдеру говорит, что ниже основных полей могут быть доступны дополнительные параметры модели:

- `Max Output Tokens`
- `Context Window`
- `Image Support`
- `Computer Use`
- `Input Price`
- `Output Price`

Эти поля не являются обязательными для первого подключения. Для базового старта достаточно корректно указать provider, base URL, key и model ID.

Отдельно на вашем скриншоте виден переключатель `Use different models for Plan and Act modes`. Это полезная возможность, если вы хотите:

- использовать более сильную модель для `Plan`
- использовать более быструю или более дешевую модель для `Act`

Но для первого запуска лучше сначала добиться стабильной работы на одной модели, а уже потом разделять режимы.

## Как выбрать model ID для Cline

У Cline нет одного универсального лучшего model ID для всех задач. Но для практической работы стоит держать в голове несколько правил.

- Берите модель, которая уверенно справляется именно с coding-задачами, а не только с обычным чат-диалогом.
- Для Cline важны не только качество ответа, но и устойчивость в tool-driven workflow: чтение файлов, анализ репозитория, выполнение команд, многошаговая реализация.
- Если вы планируете длинные задачи по большому проекту, важен хороший context window.
- Если вы часто используете `Plan` и `Act`, полезно понимать стоимость и скорость модели на длинных сессиях.

На вашем скриншоте сам интерфейс Cline показывает предупреждение, что Cline использует сложные prompts и лучше работает с более сильными моделями, а менее способные модели могут вести себя хуже ожидаемого. Практически это означает простое правило:

- для первого подключения LeanToken начните с модели, которую вы уже считаете надежной в agentic coding-сценариях
- если модель отвечает, но часто путается в многошаговой работе, попробуйте более сильный coding-oriented вариант

## Первая проверка после настройки

После того как LeanToken подключен, не начинайте сразу с большой задачи. Сначала проверьте базовый контур.

1. Отправьте простой вопрос, чтобы убедиться, что Cline вообще отвечает через новый provider.
2. Попросите Cline прочитать несколько файлов в открытом репозитории и кратко объяснить структуру проекта.
3. Переключитесь в `Plan`, если хотите сначала безопасно проверить анализ без изменений файлов.
4. После этого дайте маленькую задачу в `Act`, например поправить текст, добавить небольшой guard clause или обновить комментарий.
5. Если все прошло успешно, уже потом переходите к более серьезным изменениям и командам.

Хорошие тестовые запросы для первого запуска:

- `Прочитай package.json и кратко объясни, как запускается проект`
- `Найди точку входа приложения и опиши поток инициализации`
- `Составь короткий план, как добавить простую проверку ошибки в этот модуль`

## Практический workflow для повседневной работы

После базовой настройки Cline через LeanToken наиболее удобный рабочий поток обычно выглядит так:

1. Откройте репозиторий и начните новую задачу с четкой целью.
2. Для средних и больших задач начните в `Plan`.
3. Попросите Cline найти релевантные файлы и предложить план.
4. Уточните ограничения: стиль кода, тесты, архитектурные требования, что менять нельзя.
5. Переключитесь в `Act`, когда план вас устраивает.
6. После завершения сравните результат, при необходимости откатитесь через checkpoints и повторите итерацию.

Если задача большая, стоит дополнительно использовать:

- `Memory Bank`, чтобы сохранять контекст проекта между сессиями
- `Rules`, чтобы один раз зафиксировать conventions и ограничения
- `Workflows`, если у вас есть повторяемые процедуры
- `Hooks`, если нужно автоматически проверять или блокировать опасные действия
- `MCP servers`, если вы хотите дать Cline доступ к внешним системам

## Как уменьшить стоимость и шум в контексте

Одна из самых частых практических проблем в Cline — не сама модель, а лишний контекст. Официальная документация отдельно рекомендует использовать `.clineignore`, чтобы Cline не тратил контекст на файлы, которые не помогают решать задачу.

В `.clineignore` почти всегда стоит исключить:

- `node_modules/`
- `dist/`
- `build/`
- `.next/`
- `coverage/`
- `.env` и похожие секретные env-файлы
- большие бинарные assets, дампы данных и generated files

Это полезно сразу по трем причинам:

- снижается стоимость запросов
- ответы приходят быстрее
- модель реже вытесняет важный контекст второстепенными файлами

Если у вас большой проект, настройка `.clineignore` часто дает больший эффект, чем просто смена модели.

## Безопасность и контроль

Cline дает много мощности, поэтому важна дисциплина в настройке разрешений.

- Не включайте агрессивный auto-approve на незнакомом проекте.
- Не включайте `YOLO mode`, если не готовы к полностью автоматическому выполнению действий без подтверждения.
- Для серьезной работы начинайте с ручного подтверждения редактирования файлов и команд.
- Если хотите ускориться, включайте auto-approve постепенно и только на тех действиях, где реально понимаете риск.
- Держите checkpoints включенными, если активно экспериментируете с реализацией.
- Проверяйте MCP servers и Hooks так же строго, как проверяли бы любой внешний code execution path.

Для LeanToken это дополняется обычными правилами безопасности ключей:

- не храните API key в репозитории
- не вставляйте ключ в `.env.example`, README и issue-трекер
- не передавайте ключ в скриншотах и shared notes
- если ключ засветился, перевыпустите его

## Типичные проблемы и что проверить

### Cline не отвечает после настройки

Проверьте:

- выбран ли `API Provider = OpenAI Compatible`
- правильно ли указан `Base URL`
- не потерян ли `/v1` в endpoint
- корректен ли API key
- доступен ли указанный model ID через ваш LeanToken аккаунт

### Ошибка `Invalid API Key` или похожая ошибка авторизации

- скопируйте ключ заново из LeanToken
- убедитесь, что вставили именно API key, а не другой токен
- если есть подозрение на утечку или поврежденную конфигурацию, создайте новый key и замените старый

### Ошибка `Model Not Found`

- проверьте точный model ID в вашем LeanToken аккаунте
- убедитесь, что выбранный model ID действительно доступен через ваш endpoint
- если сомневаетесь, начните с уже известной рабочей модели и только потом переходите к экспериментам

### Слишком дорогие или слишком длинные сессии

- добавьте `.clineignore`
- начинайте новые задачи вместо бесконечного продолжения одной переписки
- используйте `Plan` до `Act`, чтобы не тратить expensive tool calls раньше времени
- при необходимости настройте разные модели для `Plan` и `Act`

### Модель отвечает, но плохо выполняет инженерные задачи

Это не редкость. Для Cline важно не просто наличие ответа, а качество agentic поведения. Если модель:

- теряет нить после нескольких шагов
- плохо следует плану
- слабо работает с командами и файловым контекстом
- выдает нестабильные правки

попробуйте более сильный coding-oriented model ID в LeanToken.

## Где Cline особенно хорошо сочетается с LeanToken

Связка Cline и LeanToken особенно удобна в таких сценариях:

- один и тот же OpenAI-compatible endpoint используется сразу в нескольких инструментах команды
- нужно централизованно менять model ID без изменения общего подхода в IDE
- важно разделить editor workflow и провайдерский уровень доступа к моделям
- вы хотите использовать BYOK не только в Cline, но и в других сервисах из вашей AI-инфраструктуры

В этом смысле LeanToken становится не просто местом, где вы храните ключ, а общей точкой управления модельным доступом для developer tools.
