# MiroFish

![Логотип MiroFish](../../images/mirofish/mirofish.png)

## О сервисе

MiroFish — это open-source движок многоагентного AI-прогнозирования. По сохраненным материалам проект строит цифровую модель общества на основе входных данных, например новостей, законопроектов, аналитических отчетов или финансовых сигналов, а затем запускает внутри нее симуляцию поведения тысяч автономных агентов.

Каждый агент получает роль, поведенческую логику, долгосрочную память и связи с другими сущностями. В результате MiroFish не просто отвечает на один prompt, а прогоняет целый сценарий: строит граф знаний, генерирует окружение, моделирует социальные взаимодействия, собирает отчет и позволяет отдельно общаться с агентами и ReportAgent после завершения симуляции.

По сохраненным источникам сервис сочетает:

- фронтенд на Vue.js 3, Vite, D3.js и Tailwind CSS
- backend на Python 3.11+ и Flask
- симуляционный движок OASIS от CAMEL-AI
- память и GraphRAG-слой через Zep Cloud
- OpenAI-compatible LLM orchestration, куда подходит и BYOK через `leantoken.tech`

![Сохраненный скриншот интерфейса MiroFish из материалов проекта](../../images/mirofish/mirofish-overview.webp)

С точки зрения конечного пользователя MiroFish ближе не к обычному AI-чату, а к цифровой песочнице для моделирования: вы задаете исходные материалы и цель прогноза, а затем наблюдаете, как цифровое общество реагирует на событие и к чему это может привести.

## Как работает MiroFish

В сохраненной документации MiroFish описан как пятишаговый workflow:

1. `Graph Building` — сервис извлекает сущности и связи из исходных материалов, строит GraphRAG и инжектирует коллективную память.
2. `Environment Setup` — генерирует персоны агентов, роли и параметры симулируемых платформ.
3. `Simulation Execution` — запускает параллельные взаимодействия агентов и обновляет их память между раундами.
4. `Report Generation` — ReportAgent анализирует логи симуляции и готовит структурированный прогнозный отчет.
5. `Deep Interaction` — после завершения можно общаться с отдельными агентами или с ReportAgent и уточнять детали сценария.

По сохраненным материалам интерфейс MiroFish построен как пошаговый wizard. Фронтенд обычно работает на `http://localhost:3000`, а backend API — на `http://localhost:5001`.

## Сценарии использования AI

По README, DeepWiki и сохраненной статье MiroFish особенно полезен в таких задачах:

- моделирование общественной реакции на новости, политические инициативы, регуляторные изменения и кризисные инфоповоды
- тестирование PR- и коммуникационных стратегий до реального запуска, когда важно заранее увидеть коалиции, конфликты и точки напряжения
- работа с финансовыми и рыночными сигналами, где нужно не просто краткое summary, а симуляция дальнейших реакций участников
- исследование социальных платформ и дискуссий через многослойную модель агентов, а не через один универсальный ответ модели
- творческие и narrative-сценарии: альтернативные сюжетные линии, продолжения историй и исследование "что будет, если"

Важно: в сохраненных материалах MiroFish позиционируется как prediction engine и digital sandbox, но там же честно отмечено, что это ранняя версия, без подтвержденных бэктестов в документации. Поэтому использовать сервис лучше как инструмент сценарного анализа и rehearsal, а не как обещание точного предсказания реальности.

## Что понадобится

Для собственного запуска MiroFish с BYOK по сохраненным материалам нужны:

- Node.js 18+
- Python 3.11+; в сохраненном README отдельно фигурирует диапазон 3.11-3.12
- `uv` для Python-зависимостей
- `npm`
- аккаунт на `https://leantoken.tech`
- API key LeanToken
- OpenAI-compatible endpoint LeanToken, если вы подключаете MiroFish через внешний провайдер
- идентификатор модели, доступной через ваш endpoint
- отдельный `ZEP_API_KEY` для Zep Cloud, потому что память и GraphRAG в MiroFish вынесены в отдельный сервис

Если вы предпочитаете контейнерный запуск, в сохраненных материалах также описан Docker-путь через `docker compose`.

## BYOK на leantoken.tech

Для MiroFish BYOK — это не экран в UI, а конфигурация LLM-провайдера в корневом `.env` файла проекта. Сохраненные материалы прямо указывают, что сервис поддерживает любой LLM API в OpenAI-compatible формате и использует такие переменные:

- `LLM_API_KEY`
- `LLM_BASE_URL`
- `LLM_MODEL_NAME`

Именно сюда можно подставить key, endpoint и модель из `leantoken.tech`.

Практически это означает следующее:

- `LLM_API_KEY` — ваш API key LeanToken
- `LLM_BASE_URL` — OpenAI-compatible endpoint, например `https://api.leantoken.tech/v1`
- `LLM_MODEL_NAME` — модель, доступная на вашем LeanToken endpoint

При этом BYOK через LeanToken закрывает только LLM-часть MiroFish. По сохраненным материалам сервис все равно требует отдельный `ZEP_API_KEY`, потому что GraphRAG и долговременная память агентов идут через Zep Cloud.

BYOK через `leantoken.tech` удобен для MiroFish, когда вы хотите:

- подключить OpenAI-compatible модель без жесткой привязки к исходному рекомендованному провайдеру
- управлять ключом и endpoint централизованно
- менять модель без переписывания логики MiroFish
- переиспользовать тот же ключ в других OpenAI-compatible инструментах

## Как получить API key на leantoken.tech

Перед настройкой MiroFish получите key и endpoint в LeanToken.

![Публичная страница LeanToken с Sign up, Log in и Dashboard](../../images/leantoken/leantoken.tech.1.landing.png)

1. Откройте `https://leantoken.tech`.
2. Нажмите `Sign up`, если у вас еще нет аккаунта, `Log in`, если аккаунт уже есть, или `Dashboard`, если вы уже вошли в систему.

![Панель LeanToken с API endpoint и быстрыми ссылками](../../images/leantoken/leantoken.tech.2.dashboard.png)

3. После входа откройте dashboard или обзорную страницу API.
4. Скопируйте API endpoint. На скриншотах показан `https://api.leantoken.tech/v1`.
5. Перейдите в `Manage keys` или `API Keys`.

![Страница API Keys в LeanToken с действием Create key](../../images/leantoken/leantoken.tech.3.key-managment.png)

6. Нажмите `Create key`.
7. Скопируйте созданный API key.
8. Подготовьте также точный идентификатор модели, которую будете использовать в MiroFish через `LLM_MODEL_NAME`.
9. Сохраните key и endpoint как секреты. Если ключ был раскрыт, перевыпустите его и замените в `.env`.

## Как добавить BYOK

Ниже поток, который лучше всего совпадает с сохраненными материалами MiroFish.

### 1. Подготовьте проект и `.env`

Сохраненный README рекомендует такой старт:

```bash
git clone https://github.com/666ghj/MiroFish.git
cd MiroFish
cp .env.example .env
```

После этого откройте `.env` и заполните LLM-блок значениями из LeanToken:

```env
LLM_API_KEY=ваш_ключ_leantoken
LLM_BASE_URL=https://api.leantoken.tech/v1
LLM_MODEL_NAME=идентификатор_модели
ZEP_API_KEY=ваш_ключ_zep
```

Что здесь важно:

- `LLM_API_KEY` и `LLM_BASE_URL` берутся из LeanToken
- `LLM_MODEL_NAME` должен совпадать с моделью, которую реально отдает ваш LeanToken endpoint
- `ZEP_API_KEY` нужно получить отдельно, потому что LeanToken не заменяет Zep Cloud memory layer

В сохраненной конфигурации также есть optional `Boost`-блок для ускорения некоторых параллельных операций. По присланному `.env`-фрагменту он использует такие переменные:

```env
LLM_BOOST_API_KEY=your_api_key_here
LLM_BOOST_BASE_URL=your_base_url_here
LLM_BOOST_MODEL_NAME=your_model_name_here
```

Если хотите использовать отдельную ускоряющую конфигурацию через LeanToken, можно заполнить этот блок тем же endpoint и той же моделью или выделить под Boost отдельную модель. Если ускоряющая конфигурация не нужна, в присланном фрагменте прямо рекомендуется не оставлять эти поля в `.env`.

### 2. Установите зависимости

Сохраненные материалы рекомендуют единый установочный сценарий:

```bash
npm run setup:all
```

Если вам удобнее раздельно:

```bash
npm run setup
npm run setup:backend
```

По сохраненной документации эта схема устанавливает root- и frontend-зависимости через `npm`, а backend-зависимости — через `uv`.

### 3. Запустите MiroFish

Для локальной разработки используйте:

```bash
npm run dev
```

После запуска по сохраненным материалам сервисы доступны так:

- фронтенд: `http://localhost:3000`
- backend API: `http://localhost:5001`

Если нужно запускать части отдельно:

```bash
npm run backend
npm run frontend
```

### 4. Пройдите пошаговый workflow в интерфейсе

После запуска откройте фронтенд и пройдите стандартные этапы MiroFish:

1. На этапе `Graph Construction` загрузите seed-материалы: новости, аналитические отчеты, policy drafts или другие входные документы.
2. На этапе `Environment Setup` проверьте, что окружение и персоны агентов собираются без LLM-auth ошибок.
3. На этапе `Simulation Execution` запустите симуляцию.
4. На этапе `Report` дождитесь итогового аналитического отчета.
5. На этапе `Interaction` задайте уточняющие вопросы отдельным агентам или ReportAgent.

Важный нюанс MiroFish: вы не вставляете LeanToken key в отдельное поле внутри wizard. По сохраненным материалам LLM-конфигурация читается из `.env`, поэтому главный индикатор корректного BYOK — успешная работа шагов, где нужна модель.

![Сохраненный скриншот демо-сцены MiroFish из материалов проекта](../../images/mirofish/mirofish-demo-world.webp)

### 5. Начните с короткой и дешевой симуляции

В сохраненном README отдельно отмечено, что симуляции могут быстро расходовать LLM-кредиты. Там же есть практическая рекомендация начинать с запусков меньше чем на 40 раундов.

Для первичной проверки BYOK лучше:

- взять небольшой набор исходных материалов
- запустить короткую симуляцию
- убедиться, что graph building и persona generation проходят без ошибок аутентификации
- только потом увеличивать сложность сценария и число раундов

## Альтернатива: Docker

Сохраненные материалы MiroFish отдельно описывают контейнерный запуск:

```bash
cp .env.example .env
docker compose up -d
```

По DeepWiki и README:

- используется образ `ghcr.io/666ghj/mirofish:latest`
- `docker compose` читает `.env` из корня проекта
- наружу мапятся порты `3000` и `5001`
- `./backend/uploads` пробрасывается в контейнер, чтобы сохранять загруженные файлы и логи симуляции

Для BYOK-логики Docker-путь ничем не отличается от source deployment: главное, чтобы `.env` уже содержал LeanToken key, endpoint и модель.

## Практические замечания

- По сохраненной статье MiroFish — ранняя версия `0.1.0`, и в этих материалах нет подтвержденных бэктестов.
- В статье отдельно указано, что на Windows проект пока не был полноценно проверен.
- В документации MiroFish рекомендует OpenAI-compatible LLM-конфигурацию и отдельно использует Zep Cloud, поэтому один только LeanToken key не закрывает всю инфраструктуру сервиса.
- Не храните `.env` в git и не публикуйте `LLM_API_KEY`, `ZEP_API_KEY` и endpoint в логах или публичных заметках.
- Если вы хотите сначала понять сам продукт, а потом поднимать локальную инсталляцию, в сохраненных материалах есть [онлайн-демо](https://666ghj.github.io/mirofish-demo/).

## Что проверить после настройки

- `.env` содержит корректные `LLM_API_KEY`, `LLM_BASE_URL`, `LLM_MODEL_NAME` и `ZEP_API_KEY`
- MiroFish стартует без ошибок через `npm run dev` или `docker compose up -d`
- фронтенд доступен на `localhost:3000`, backend — на `localhost:5001`
- шаги graph building и environment setup не падают на аутентификации LLM
- симуляция запускается и возвращает отчет
- вы можете открыть post-simulation interaction и задать вопрос агенту или ReportAgent
- ключи LeanToken и Zep не попали в git, shell history или shared configs
