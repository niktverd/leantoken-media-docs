# Roo Code

## О сервисе

Roo Code — это open source AI-coding suite для разработки с помощью LLM. Официально сервис доступен в двух основных форматах:

- как локальное расширение для VS Code и совместимых редакторов
- как Roo Code Cloud Agents для автономной работы через web, Slack, GitHub и другие интеграции

Главная идея Roo Code — model-agnostic подход. Сам Roo не является моделью: ему нужен inference provider, а вы сами выбираете провайдера, model ID, API key и дополнительные настройки. Это важно для BYOK-сценария, потому что Roo Code не привязывает вас к одному облаку или одной подписке.

Практически Roo Code умеет гораздо больше, чем обычный AI-чат в редакторе. По официальной документации он может:

- генерировать код по текстовому описанию
- объяснять структуру и поведение существующей кодовой базы
- рефакторить и исправлять ошибки
- обновлять документацию
- запускать команды
- работать с MCP servers
- вести многошаговый workflow с разными режимами работы

Эта страница сфокусирована на локальном расширении Roo Code, потому что именно там путь BYOK через `leantoken.tech` и OpenAI-compatible endpoint максимально прямой.

## Когда Roo Code особенно полезен

Roo Code хорошо подходит для таких сценариев:

- разбор незнакомого репозитория, когда нужно быстро понять архитектуру, точки входа и зависимости
- исправление багов через связку `@problems`, `@terminal`, чтение файлов и запуск команд
- реализация небольших и средних задач по ТЗ без постоянного ручного переключения между файлами
- рефакторинг и подготовка pull request с последовательной работой по шагам
- генерация тестов, changelog, release notes и технической документации
- обзор UI-скриншотов и изображений, если выбранная модель поддерживает vision
- multi-step задачи, где удобно разделять planning, coding, debug и orchestration между разными режимами

Если говорить проще, Roo Code полезен там, где одного autocomplete уже мало, а нужен именно агентный workflow с доступом к файлам, командам и контексту проекта.

## Форматы работы Roo Code

Прежде чем настраивать BYOK, полезно понимать разницу между двумя основными формами продукта.

### 1. Roo Code VS Code Extension

Это локальный сценарий. Roo Code работает прямо в вашем редакторе, использует текущую рабочую директорию, предлагает инструменты, просит разрешения на действия и позволяет вам контролировать каждый шаг.

Именно этот формат лучше всего подходит для BYOK через LeanToken, потому что вы напрямую указываете:

- провайдера
- base URL
- API key
- model ID
- параметры модели и профиля

### 2. Roo Code Cloud Agents

Это отдельный облачный формат для автономной работы агентной команды через web и внешние интеграции. Он полезен для параллельной работы, задач из Slack и GitHub и командных процессов, но поток настройки через LeanToken в этом документе не рассматривается.

## Что понадобится

Для настройки Roo Code через BYOK обычно нужны:

- VS Code `1.84.0` или новее, либо совместимый редактор
- установленное расширение Roo Code
- аккаунт на `https://leantoken.tech`
- API key LeanToken
- OpenAI-compatible endpoint `https://api.leantoken.tech/v1`
- модель, которая доступна через ваш LeanToken endpoint и поддерживает OpenAI-compatible native tool calling

Последний пункт критичен. Roo Code в официальной документации прямо указывает, что использует только native tool calling. XML-based fallback нет. Это означает, что модель может отвечать на обычные чат-запросы, но все равно быть плохим выбором для Roo Code, если она не умеет корректно работать с инструментами.

## BYOK на leantoken.tech

В контексте Roo Code BYOK на `leantoken.tech` означает, что вы:

1. получаете собственный API key в LeanToken
2. подключаете LeanToken как `OpenAI Compatible` provider
3. указываете `https://api.leantoken.tech/v1` в качестве Base URL
4. выбираете модель, доступную через ваш endpoint
5. используете Roo Code как обычный coding agent в редакторе, но уже поверх собственного ключа и собственного модельного доступа

Это удобно по нескольким причинам:

- вы сами управляете ключом и endpoint
- вы можете менять model ID без смены основного workflow в IDE
- один и тот же LeanToken key можно использовать в других OpenAI-compatible интеграциях
- вы не зависите только от Router-пути или от конкретной first-party подписки

Если вам нужен самый простой путь без API key, Roo Code рекомендует Roo Code Router. Но если задача именно в BYOK и в управлении своим endpoint, то для LeanToken нужен вариант `OpenAI Compatible`.

## Как получить API key на leantoken.tech

Перед настройкой Roo Code получите key и endpoint в LeanToken.

![Публичная страница LeanToken с Sign up, Log in и Dashboard](../../images/leantoken/leantoken.tech.1.landing.png)

1. Откройте `https://leantoken.tech`.
2. Нажмите `Sign up`, если у вас еще нет аккаунта, `Log in`, если аккаунт уже есть, или `Dashboard`, если вы уже вошли в систему.

![Панель LeanToken с API endpoint и быстрыми ссылками](../../images/leantoken/leantoken.tech.2.dashboard.png)

3. После входа откройте dashboard или обзорную страницу API.
4. Скопируйте API endpoint. На скриншотах показан `https://api.leantoken.tech/v1`.
5. Перейдите в раздел управления ключами через `Manage keys` или `API Keys`.

![Страница API Keys в LeanToken с действием Create key](../../images/leantoken/leantoken.tech.3.key-managment.png)

6. Нажмите `Create key`.
7. Скопируйте созданный API key.
8. Если планируете несколько сценариев работы, заранее определите, будете ли использовать один общий key или разные keys для разных IDE/проектов.
9. Держите key в секрете. Если он попал в git, shell history, логи, issue или скриншот, перевыпустите его.

## Как установить Roo Code

Официальная документация Roo Code рекомендует два основных пути установки.

### Через VS Code Marketplace

1. Откройте VS Code.
2. Перейдите в `Extensions`.
3. Найдите `Roo Code`.
4. Выберите расширение от `RooVeterinaryInc`.
5. Нажмите `Install`.
6. Если редактор попросит перезагрузку, подтвердите reload.

После установки в activity bar появится иконка Roo Code.

### Через Open VSX

Этот путь полезен для редакторов без доступа к Marketplace, например некоторых VS Code-compatible fork-ов.

1. Откройте раздел расширений в редакторе.
2. Найдите `Roo Code`.
3. Установите расширение.
4. Перезагрузите редактор, если это требуется.

### Установка из VSIX

Если вам нужен ручной путь:

1. Скачайте `.vsix` из официальных GitHub Releases Roo Code.
2. В VS Code откройте меню расширений.
3. Выберите `Install from VSIX...`.
4. Укажите скачанный файл.

## Как настроить LeanToken в Roo Code

Ниже основной рабочий сценарий для Roo Code и LeanToken.

![Экран Providers в Roo Code Settings с профилем default, OpenAI Compatible, base URL LeanToken и выбранной моделью](../../images/roocode/roocode.png)

На вашем скриншоте уже показан именно тот экран, который нужен для BYOK:

- раздел `Providers`
- профиль `default`
- `API Provider = OpenAI Compatible`
- `Base URL = https://api.leantoken.tech/v1`
- заполненное поле `API Key`
- выбранная модель `qwen3-235b`

### Пошаговая настройка

1. Откройте панель Roo Code в VS Code.
2. Нажмите на значок шестеренки, чтобы открыть `Settings`.
3. Перейдите на вкладку `Providers`.
4. В поле `Configuration Profile` оставьте `default` или создайте новый профиль через кнопку `+`.
5. В поле `API Provider` выберите `OpenAI Compatible`.
6. В поле `Base URL` укажите `https://api.leantoken.tech/v1`.
7. В поле `API Key` вставьте ваш LeanToken key.
8. В поле `Model` выберите нужный model ID.
9. Нажмите `Save`.

Минимальный рабочий набор параметров выглядит так:

```text
API Provider: OpenAI Compatible
Base URL: https://api.leantoken.tech/v1
API Key: <ваш_LeanToken_API_key>
Model: <ваш_model_id>
```

### Что означают дополнительные поля на экране

По документации Roo Code для OpenAI-compatible провайдера и по вашему скриншоту интерфейс может показывать дополнительные model settings и metadata:

- `Max Output Tokens`
- `Context Window`
- `Image Support`
- `Computer Use`
- `Input Price`
- `Output Price`

На вашем скриншоте Roo Code также показывает служебные переключатели и provider-specific параметры, например:

- `Enable streaming`
- `Use legacy OpenAI API format`
- `Include max output tokens`
- дополнительные параметры для отдельных семейств моделей

Это значит, что после выбора модели стоит проверить не только сам факт подключения, но и то, как Roo интерпретирует возможности конкретного model ID.

### Практическая рекомендация по выбору модели

Если вы впервые подключаете LeanToken к Roo Code:

1. начните с модели, которую вы уже проверяли в tool-driven задачах
2. не оценивайте модель только по качеству чат-ответа
3. отдельно проверьте чтение файлов, запуск инструментов и устойчивость многошагового workflow

Если модель формально отвечает, но плохо вызывает инструменты, реальная польза от Roo Code будет заметно ниже.

## Профили конфигурации

Одна из самых полезных возможностей Roo Code для BYOK-сценария — `API Configuration Profiles`.

Профили позволяют хранить несколько наборов настроек одновременно. У каждого профиля могут быть свои:

- provider settings
- API keys
- model selection
- temperature и похожие параметры
- rate limit settings
- provider-specific опции

Это особенно удобно, если вы хотите:

- держать отдельный профиль под LeanToken
- использовать разные модели для разных типов задач
- быстро переключаться между провайдерами без постоянного ручного редактирования полей

### Как использовать профили на практике

Для Roo Code с LeanToken удобна такая схема:

- `default` или `leantoken-main` для основного coding workflow
- отдельный профиль для более дешевой или быстрой модели
- отдельный профиль для vision-задач, если у вас есть совместимая модель

В официальной документации Roo Code также отмечает:

- профили можно создавать через кнопку `+` рядом с селектором
- профили можно переключать не только в `Settings`, но и через dropdown в самом chat interface
- часто используемые профили можно pin-ить, чтобы они были сверху списка
- их можно переименовывать через иконку карандаша
- их можно удалять через корзину
- задачи запоминают профиль, с которым были начаты
- разные mode-режимы могут быть привязаны к разным профилям

Это снижает риск случайного переключения модели посреди длинной задачи.

### Где Roo хранит ключи

По документации Roo Code API keys хранятся в `VSCode Secret Storage` и не должны показываться как plain text в обычных настройках. Это не отменяет базовые правила безопасности, но лучше, чем хранить ключ напрямую в проектных файлах.

## Режимы Roo Code

Roo Code строится вокруг режимов работы. Они задают поведение модели, допустимые инструменты и стиль выполнения задачи.

### Code

Основной режим по умолчанию.

Подходит для:

- написания кода
- рефакторинга
- исправления багов
- большинства повседневных задач разработки

У режима полный доступ к `read`, `edit`, `command` и `mcp`.

### Ask

Режим для объяснений и разбора контекста без прав на изменение файлов и запуск команд.

Подходит для:

- объяснения незнакомого кода
- обсуждения архитектуры
- проверки идей
- безопасного чтения проекта без риска случайных правок

### Architect

Режим для проектирования и planning-задач.

Подходит для:

- разбиения большой задачи на шаги
- системного дизайна
- проектирования изменений перед реализацией

### Debug

Режим для диагностики и поиска причин проблем.

Подходит для:

- падений тестов
- ошибок сборки
- разборов логов
- поиска причины нестабильного поведения

### Orchestrator

Режим для декомпозиции больших задач и делегирования подзадач другим режимам.

Подходит для:

- длинных feature-задач
- крупных рефакторингов
- цепочек задач, где нужны planning, coding и debugging по очереди

У этого режима нет прямого tool access к обычным операциям с файлами и командами. Вместо этого он делегирует подзадачи другим режимам.

### Как переключать режимы

Roo Code документирует четыре основных пути переключения:

1. dropdown слева от chat input
2. slash commands вроде `/code`, `/ask`, `/architect`, `/debug`, `/orchestrator`
3. горячая клавиша `Cmd + .` на macOS или `Ctrl + .` на Windows/Linux
4. предложения смены режима, которые Roo иногда показывает сам

### Почему это важно для BYOK

Roo Code запоминает последний model selection для режима. Это позволяет, например:

- держать более сильную reasoning-модель для `Architect`
- более быструю модель для `Code`
- отдельный профиль для `Debug`

Для LeanToken это полезно, если вы хотите управлять стоимостью и качеством на уровне конкретного workflow, а не одним универсальным model ID для всего подряд.

## Как Roo Code использует инструменты

Официальная документация Roo Code описывает инструментальную модель очень прямо: Roo не просто отвечает текстом, а вызывает инструменты для работы с кодом и средой.

Основные категории:

- `read` для чтения и поиска по файлам
- `edit` для создания и изменения файлов
- `command` для выполнения команд
- `image` для AI-генерации изображений
- `mcp` для интеграций через Model Context Protocol
- workflow-инструменты для follow-up вопросов, завершения задачи, смены режима и новых подзадач

### Как выглядит workflow

Типичный сценарий такой:

1. вы описываете задачу естественным языком
2. Roo выбирает нужный инструмент
3. показывает параметры инструмента
4. ждет вашего подтверждения
5. выполняет действие
6. продолжает работу, пока задача не будет завершена

По умолчанию это безопаснее, чем полностью автономный режим, потому что вы контролируете:

- какие файлы будут изменены
- какие команды будут запущены
- какие действия действительно дойдут до исполнения

## Почему native tool calling критичен

Для Roo Code это один из ключевых технических моментов.

В документации `OpenAI Compatible` Roo Code прямо сказано:

- Roo использует только native tool calling
- XML-based fallback не поддерживается
- если модель не поддерживает OpenAI-compatible function/tool calling, она не подходит для Roo Code

Из этого следует практический вывод для LeanToken:

- совместимость с OpenAI-compatible chat endpoint сама по себе еще не гарантирует нормальную работу Roo Code
- нужно отдельно проверить совместимость выбранной модели с native tool calling
- если модель дает ошибки tool calling, лучше сменить model ID, чем пытаться «дожать» неподходящую конфигурацию

## Полезные mentions в Roo Code

Одна из сильных сторон Roo Code — контекстные mentions. Они позволяют подтягивать не только обычный текст, но и файлы, папки, изображения, diagnostics и git-контекст.

### Самые полезные mentions для практики

- `@/path/to/file.ts` — добавить файл в контекст
- `@/path/to/folder/` — добавить содержимое файлов из директории
- `@/path/to/image.png` — добавить изображение, если модель поддерживает vision
- `@problems` — подтянуть diagnostics из VS Code Problems
- `@terminal` — подтянуть последний command output
- `@git-changes` — добавить незакоммиченные изменения
- `@https://example.com` — подтянуть содержимое веб-страницы

### Реальные примеры запросов

```text
@problems Исправь все ошибки сборки и объясни корневую причину.
```

```text
Объясни, что делает @/src/app.ts и какие здесь есть риски архитектуры.
```

```text
Сравни @git-changes с текущей задачей и предложи хороший commit message.
```

```text
Что не так с этим экраном? @/screenshots/bug.png
```

Для Roo Code и LeanToken это особенно полезно, если вы хотите получать не абстрактные советы, а ответы на базе реального контекста репозитория, терминала и текущих изменений.

## Дополнительная возможность: Codebase Indexing

Roo Code умеет строить semantic index кодовой базы. По официальной документации этот механизм:

1. разбирает код через Tree-sitter
2. создает embeddings для semantic blocks
3. хранит вектора в Qdrant
4. дает Roo доступ к `codebase_search`

Это отдельная возможность, не обязательная для базовой настройки BYOK. Но она полезна для больших репозиториев, где обычного file-by-file чтения уже мало.

Важно понимать архитектуру этой функции:

- нужен embedding provider
- нужна vector database, обычно Qdrant
- если вы хотите использовать OpenAI-compatible embedder, у вас должен быть совместимый endpoint и совместимая embedding-модель

Для LeanToken это означает, что indexing стоит настраивать отдельно от основного chat/profile path. Сначала лучше добиться стабильной работы обычного Roo workflow, а уже потом добавлять embeddings и vector search.

## Практический стартовый workflow

После настройки BYOK не начинайте с большой feature-задачи. Гораздо безопаснее сделать короткую проверку по шагам.

### Шаг 1. Проверка чтения контекста

Попросите Roo прочитать один-два файла:

```text
Объясни структуру этого проекта и начни с @/package.json и @/src/
```

### Шаг 2. Проверка диагностики

Если в проекте есть ошибки:

```text
@problems Разбери ошибки и предложи порядок исправления.
```

### Шаг 3. Проверка инструментов

Дайте небольшую безопасную задачу:

```text
Добавь краткий комментарий в этот файл и покажи, какие изменения ты собираешься внести.
```

### Шаг 4. Проверка команд

Если вы готовы разрешить команду:

```text
Запусти тесты для этого пакета и кратко объясни, что сломалось.
```

Так вы быстро поймете:

- видит ли Roo файлы
- корректно ли модель вызывает tools
- стабильно ли работает provider path через LeanToken

## Безопасность и эксплуатация

Roo Code по умолчанию строится вокруг approvals, и это хорошая модель безопасности.

### Что стоит оставить включенным по умолчанию

- ручное подтверждение правок
- ручное подтверждение команд
- аккуратная проверка всех tool proposals

### Что нужно понимать про Auto-Approve

В документации Roo Code есть прямое предупреждение: auto-approve ускоряет работу, но существенно повышает риски. Для командной строки и изменений файлов это может привести к потере данных, поломке файлов или другим нежелательным последствиям.

Для LeanToken-сценария практическая рекомендация простая:

- сначала добейтесь стабильного ручного workflow
- только потом включайте selective auto-approve
- не включайте агрессивный auto-approve на незнакомых проектах и production-конфигурациях

### Базовые правила безопасности

- не храните LeanToken key в репозитории
- не вставляйте ключ в проектные конфиги без необходимости
- не публикуйте скриншоты с незамаскированным ключом
- при утечке сразу перевыпускайте ключ
- используйте отдельные профили для разных типов работы, если хотите разграничить доступ

## Типичные проблемы и как их проверять

### Roo Code не подключается к LeanToken

Проверьте:

- правильность `Base URL`
- правильность API key
- доступность `https://api.leantoken.tech/v1`
- не обрезался ли ключ при копировании

### Roo Code видит модель, но плохо работает с инструментами

Это почти всегда повод проверить:

- поддерживает ли конкретный model ID native tool calling
- не ограничивает ли провайдер tool/function-calling semantics
- подходит ли модель для многошагового coding workflow, а не только для одноразового чата

### Ошибка `Model Not Found`

Проверьте точный model ID в вашем LeanToken endpoint. Ошибка часто связана не с ключом, а именно с именем модели.

### Ошибки соединения

Проверьте:

- нет ли опечатки в endpoint
- доступен ли endpoint из вашей сети
- не блокирует ли прокси или firewall запросы IDE

### Контекст быстро переполняется

Roo Code отдельно рекомендует разумно относиться к `Max Tokens` и context window. Если вы используете слишком большие thinking budgets, вы быстрее съедаете место под историю задачи.

Практически помогает:

- держать более высокие thinking limits для `Architect` и `Debug`
- использовать более умеренные настройки для `Code`
- разбивать работу на несколько задач вместо одной бесконечной сессии

## Что проверить после настройки

- Roo Code открывается без ошибок и виден в sidebar
- в `Providers` сохранен профиль с `OpenAI Compatible`
- `Base URL` установлен в `https://api.leantoken.tech/v1`
- модель отвечает на короткий тестовый запрос
- Roo может читать файлы и предлагать инструменты
- tool-driven операции не падают из-за несовместимости модели
- ключ хранится как секрет и не попал в git, логи и заметки

## Итоговая рекомендация

Для Roo Code и LeanToken самый надежный стартовый путь сейчас такой:

1. установить Roo Code в VS Code
2. создать отдельный профиль под LeanToken
3. выбрать `OpenAI Compatible`
4. указать `https://api.leantoken.tech/v1`
5. вставить LeanToken API key
6. выбрать модель, которая уже проверена на native tool calling
7. прогнать короткий тестовый workflow с файлами, diagnostics и командами

Если эта базовая связка работает стабильно, дальше уже можно наращивать конфигурацию:

- отдельные профили под разные режимы
- vision-сценарии
- indexing
- selective auto-approve
- custom modes и более сложные workflow
