# Kilo Code

## О сервисе

Kilo Code — это open source AI coding assistant, который помогает писать код, разбирать кодовую базу, исправлять ошибки, проводить рефакторинг и автоматизировать инженерные задачи. В актуальной документации Kilo позиционируется как инструмент, который работает там, где уже работает разработчик: в IDE, в CLI, в браузере, на мобильных устройствах и в Slack.

Для практической разработки это важно по двум причинам:

- Kilo Code не ограничивается одним встроенным способом доступа к моделям
- один и тот же workflow можно перенести между VS Code, JetBrains и CLI, не меняя базовую логику работы

Эта страница сфокусирована на самом полезном для LeanToken сценарии: подключение `https://leantoken.tech` в Kilo Code как `OpenAI Compatible` provider с собственным API key и собственным model ID.

## Где Kilo Code особенно полезен

Kilo Code полезен не только как чат с моделью, а как рабочий AI-инструмент для реальной разработки.

- быстрый разбор незнакомого репозитория с чтением файлов и объяснением архитектуры
- реализация небольших и средних задач по ТЗ прямо из чата
- последовательный рефакторинг, когда нужно сначала понять код, потом аккуратно внести правки
- отладка через связку диагностики, терминала, логов и пошаговых исправлений
- обновление документации, changelog и вспомогательных файлов рядом с кодом
- многошаговые инженерные задачи, где важно переключаться между planning, coding и debugging

## Как устроена работа в Kilo Code

Чтобы настройка BYOK имела смысл, полезно заранее понимать несколько базовых механизмов Kilo.

### Agents

В текущей документации Kilo Code использует термин `agents`. Это специализированные рабочие режимы с разным уровнем доступа к инструментам и разным фокусом.

- `code` — основной агент для реализации, изменения файлов и обычной разработки
- `ask` — режим для вопросов и объяснений без изменения кодовой базы
- `plan` — режим для проектирования, планирования и подготовки реализации
- `debug` — режим для диагностики проблем и пошагового поиска причины ошибок

Практически это означает простое правило:

- `ask` и `plan` лучше подходят для исследования и проектирования
- `code` и `debug` лучше подходят для фактической работы над задачей

### Контекст и упоминания

Kilo умеет сам искать релевантные файлы через встроенные инструменты `read`, `grep` и `glob`, но ручной контроль тоже есть. В документации отдельно описаны `@`-mentions:

- `@/path/to/file` для точного файла
- `@problems` для ошибок из панели Problems
- `@terminal` для вывода терминала
- `@git-changes` для незакоммиченных изменений

Кроме этого, расширение автоматически подмешивает контекст активного файла и открытых вкладок редактора. Поэтому Kilo хорошо подходит для задач, где нужно быстро перейти от вопроса к точечному исправлению.

### Подтверждения действий и Auto-Approve

По текущему quickstart Kilo Code работает итеративно: агент предлагает действие, вы его подтверждаете или отклоняете, затем работа продолжается дальше.

Документация отдельно отмечает, что по умолчанию большая часть инструментов auto-approved, а подтверждение особенно важно для:

- shell-команд
- доступа к внешним директориям
- чувствительных чтений файлов

Если нужен более строгий режим, в `Settings -> Auto-Approve` можно выставить для каждого инструмента один из вариантов:

- `Allow`
- `Ask`
- `Deny`

Для BYOK-сценария это важно, потому что сильная модель через LeanToken может работать довольно автономно. Чем мощнее модель и чем выше уровень доверия к ней, тем внимательнее нужно отнестись к разрешениям на `bash`, редактирование файлов и внешние инструменты.

### Настройки и конфиги

Текущая документация Kilo говорит, что VS Code extension и CLI используют одну и ту же базовую систему настроек. Изменения, внесенные через UI, отражаются и в конфигурационных файлах.

Основные места хранения конфигурации:

- глобальный конфиг: `~/.config/kilo/kilo.jsonc`
- проектный конфиг: `kilo.jsonc` в корне проекта
- альтернативный проектный путь: `.kilo/kilo.jsonc`

Это полезно для командной работы и для CLI, но создает и риск: если вы коммитите конфиги в git, нельзя оставлять там ключи и другие секреты.

## Почему Kilo Code удобен для BYOK

Kilo Code можно использовать через встроенный провайдер Kilo, но это не единственный путь. В актуальной документации отдельно указано, что сервис поддерживает десятки внешних providers, включая `OpenAI Compatible`. Это и делает его удобным для LeanToken.

## BYOK на leantoken.tech: что это значит в Kilo Code

В контексте Kilo Code Bring Your Own Key на `leantoken.tech` означает, что вы:

1. получаете собственный API key в LeanToken
2. подключаете LeanToken в Kilo как отдельный provider
3. выбираете `OpenAI Compatible`
4. указываете endpoint `https://api.leantoken.tech/v1`
5. вставляете собственный key
6. выбираете подходящий model ID под свой workflow

По официальной документации Kilo Code для OpenAI-compatible провайдера достаточно указать четыре основных параметра:

- `API Provider`
- `Base URL`
- `API Key`
- `Model`

Этого достаточно для стартовой настройки. Дополнительные поля вроде `Max Output Tokens`, `Context Window` или `Image Support` можно уточнить позже, когда базовое подключение уже работает стабильно.

## Что понадобится до настройки

Перед началом подготовьте:

- установленный Kilo Code в VS Code, JetBrains или другой поддерживаемой среде
- аккаунт на `https://leantoken.tech`
- LeanToken API key
- endpoint `https://api.leantoken.tech/v1`
- хотя бы один model ID, который вы хотите использовать через LeanToken

Ниже основной flow показан для UI расширения, потому что ваш скриншот относится именно к нему.

## Как получить API key на leantoken.tech

Перед настройкой Kilo Code получите API key и endpoint в LeanToken.

![Публичная страница LeanToken с Sign up, Log in и Dashboard](../../images/leantoken/leantoken.tech.1.landing.png)

1. Откройте `https://leantoken.tech`.
2. Нажмите `Sign up`, если у вас еще нет аккаунта, `Log in`, если аккаунт уже есть, или `Dashboard`, если вы уже авторизованы.

![Панель LeanToken с API endpoint и быстрыми ссылками](../../images/leantoken/leantoken.tech.2.dashboard.png)

3. После входа откройте dashboard или обзорную страницу API.
4. Скопируйте API endpoint. На текущих скриншотах показан `https://api.leantoken.tech/v1`.
5. Перейдите в раздел управления ключами через `Manage keys` или `API Keys`.

![Страница API Keys в LeanToken с действием Create key](../../images/leantoken/leantoken.tech.3.key-managment.png)

6. Нажмите `Create key`.
7. Скопируйте созданный API key.
8. Храните key как секрет. Если он попал в git, логи, историю терминала или скриншоты, перевыпустите его.

## Как установить Kilo Code

Официальная документация Kilo поддерживает несколько сред: VS Code, VS Code Preview, JetBrains, CLI, Slack и другие IDE. Но для LeanToken-страницы удобнее всего отталкиваться от VS Code-потока, потому что именно он совпадает с текущим экраном настроек на вашем скриншоте.

### Установка в VS Code

1. Откройте VS Code.
2. Перейдите в `Extensions`.
3. Найдите `Kilo Code`.
4. Нажмите `Install`.
5. После установки убедитесь, что в боковой панели появилась иконка Kilo Code.

Альтернативный путь из документации:

```bash
code --install-extension kilocode.kilo-code
```

### Что происходит после установки

Если вы хотите использовать встроенный провайдер Kilo, расширение предложит вход через аккаунт Kilo. Для LeanToken этот шаг не является обязательным: вы можете вместо этого подключить собственный provider через экран настроек.

## Как добавить LeanToken как custom provider в Kilo Code

На вашем скриншоте показан актуальный и самый полезный для LeanToken путь: вкладка `Providers` и окно `Custom provider`.

По официальной документации настройки Kilo Code открываются через значок шестеренки в панели расширения.

### Пошаговая настройка

1. Откройте панель Kilo Code в боковой панели редактора.
2. Нажмите на значок шестеренки, чтобы открыть `Settings`.
3. Перейдите на вкладку `Providers`.
4. Если нужный провайдер не виден сразу, откройте список дополнительных провайдеров и выберите путь добавления custom/OpenAI-compatible provider.
5. Откройте окно `Custom provider`, как на скриншоте.
6. В поле `Provider ID` укажите стабильный идентификатор без пробелов и заглавных букв. Практический вариант: `lean-token`.
7. В поле `Display name` укажите человекочитаемое имя, например `Lean Token`.
8. В поле `Base URL` укажите `https://api.leantoken.tech/v1`.
9. В поле `API key` вставьте ваш LeanToken API key.
10. Подтвердите подключение провайдера.
11. После успешного подключения выберите нужную модель в селекторе модели Kilo Code.
12. Отправьте короткий тестовый запрос, чтобы убедиться, что провайдер отвечает корректно.

### Что означает каждое поле в окне `Custom provider`

На этом экране важно понимать назначение полей, чтобы не ошибиться в конфигурации.

#### `Provider ID`

Это внутренний технический идентификатор провайдера внутри Kilo. Он нужен для стабильной конфигурации и обычно должен быть:

- в нижнем регистре
- без пробелов
- с использованием букв, цифр, дефисов или подчеркиваний

Для LeanToken безопаснее использовать что-то вроде:

```text
lean-token
```

#### `Display name`

Это имя, которое вы будете видеть в интерфейсе. Здесь уже можно указать более читаемый вариант:

```text
Lean Token
```

#### `Base URL`

Это главный параметр для OpenAI-compatible схемы. Для LeanToken используйте:

```text
https://api.leantoken.tech/v1
```

Kilo отдельно документирует поддержку и полных endpoint URL. Это полезно для нестандартных прокси и enterprise-схем. Но для обычного LeanToken-подключения стандартного base URL достаточно.

#### `API key`

Сюда вставляется ваш LeanToken API key. На скриншоте ниже полем также подсказано, что его можно оставить пустым, если аутентификация делается через headers. Для обычного сценария LeanToken это не нужно: API key должен быть заполнен прямо здесь.

### Минимальная рабочая конфигурация

Для большинства случаев достаточно такого набора:

```text
API Provider: OpenAI Compatible
Base URL: https://api.leantoken.tech/v1
API Key: <ваш_LeanToken_API_key>
Model: <ваш_model_id>
```

## Как выбрать model ID для Kilo Code через LeanToken

Самая частая ошибка в AI coding assistants — выбрать модель, которая неплохо отвечает в чате, но плохо ведет себя в многошаговом инженерном workflow. В документации Kilo есть полезное практическое правило: вместо фиксированного списка победителей лучше смотреть на общие принципы выбора модели.

### Практическая логика выбора

- для сложных coding-задач лучше подходят premium-модели класса Claude Sonnet/Opus, GPT-5 и сильные Gemini-подобные варианты
- для повседневной разработки обычно лучше работают mid-tier модели с хорошим балансом скорости и качества
- для бюджетного режима могут подойти современные Qwen-, DeepSeek- и похожие модели, если они стабильно справляются с вашими задачами

### Почему context window важен

По model guide Kilo размер контекста заметно влияет на удобство работы:

- `32K-64K` обычно хватает для маленьких проектов и локальных задач
- `128K` хорошо подходит для обычных приложений и нескольких связанных файлов
- `256K+` полезен для больших репозиториев и cross-file reasoning

Если вы работаете с длинными сессиями, полезно не завышать `Max Tokens` без необходимости. Документация Kilo отдельно предупреждает, что слишком большой output budget съедает место, которое иначе осталось бы под историю диалога.

### Практическая рекомендация для первого запуска

1. Возьмите модель, которую вы уже считаете надежной именно в coding-задачах.
2. Проверьте ее не только на вопросах, но и на реальной работе с файлами.
3. Если модель часто теряет нить на длинных задачах, поднимите класс модели или увеличьте доступный context window.
4. Если ответы хорошие, но работа слишком медленная и дорогая, попробуйте более быстрый mid-tier вариант.

## Как работать в Kilo после подключения LeanToken

После подключения провайдера не начинайте сразу с большой задачи на полдня. Сначала лучше проверить базовый контур.

### Первая проверка

1. Откройте Kilo Code.
2. Выберите подключенный LeanToken provider и нужную модель.
3. Отправьте короткий запрос вроде:

```text
Объясни, какие файлы в этом проекте отвечают за инициализацию приложения.
```

4. Затем дайте маленькую реальную задачу:

```text
Посмотри на @problems и предложи минимальный план исправления.
```

5. После этого проверьте editing-flow:

```text
Исправь одну конкретную ошибку из @problems и покажи diff перед изменением.
```

### Какой agent использовать

- `ask` — когда нужен ответ без правок
- `plan` — когда нужна стратегия или пошаговый план
- `code` — когда пора вносить изменения
- `debug` — когда вы уже разбираете баг, логи и терминал

Такой раздельный workflow часто дает результат лучше, чем попытка делать все подряд в одном и том же режиме.

## Полезные возможности Kilo, которые стоит знать после настройки

### 1. Custom agents

Kilo позволяет создавать собственных агентов через:

- `Settings -> Agent Behaviour -> Agents`
- markdown-файлы в `.kilo/agents/`
- записи в конфиге

Это особенно полезно, если вы хотите сделать отдельного:

- `docs-writer`
- `reviewer`
- `test-engineer`
- `security-checker`

Для LeanToken это удобно, потому что одна и та же модельная конфигурация может обслуживать сразу несколько агентных ролей.

### 2. Общие настройки между UI и CLI

Документация прямо указывает, что настройки расширения и CLI разделяют одну конфигурационную основу. Практически это означает:

- вы можете настроить провайдера в UI
- затем использовать тот же набор настроек через CLI
- и наоборот

Это хороший вариант для разработчиков, которые чередуют IDE и терминал.

### 3. Context mentions

`@`-упоминания заметно ускоряют работу:

- `@/src/app.ts` для конкретного файла
- `@terminal` для ошибок из последней команды
- `@problems` для диагностики из редактора
- `@git-changes` для обзора незакоммиченных правок

Если вы хотите получить от LeanToken-модели максимально точный ответ внутри Kilo, это один из самых полезных инструментов.

### 4. Autocomplete

Kilo имеет отдельную функцию autocomplete. В актуальной документации она описана как Fill-in-the-Middle completion на базе Codestral.

Важно понимать, что это не то же самое, что chat/provider configuration. Поэтому сначала стоит стабильно настроить именно основной provider для агента через LeanToken, а уже потом отдельно проверять, как в вашей среде ведет себя autocomplete.

### 5. Auto-Approve

Auto-Approve ускоряет работу, но Kilo сам предупреждает о риске. Если вы работаете:

- в важном продакшен-репозитории
- с секретами
- с shell-командами
- с автоматическими правками в нескольких файлах

лучше не включать избыточно широкие разрешения без необходимости.

## Безопасность и эксплуатация

Для Kilo + LeanToken есть несколько практических правил, которые действительно важны.

### Не храните ключи в git

Kilo использует конфиги `kilo.jsonc`, а документация прямо предупреждает не коммитить туда API keys и другие секреты.

### Осторожно с export/import

Документация Kilo отдельно подчеркивает, что экспорт настроек содержит API provider profiles и API keys в plaintext. Такой JSON-файл нужно считать чувствительным секретом.

### Разделяйте экспериментальные и боевые профили

Если вы работаете с несколькими моделями или разными командами, имеет смысл разделять провайдеры и конфиги так, чтобы тестовые сценарии не мешали рабочим.

### Перевыпускайте key после утечки

Если LeanToken key попал в:

- git history
- issue tracker
- screenshot
- экспорт настроек, отправленный не туда
- shell history

его нужно перевыпустить и заменить во всех конфигурациях Kilo.

## Типичные проблемы и как их проверять

Официальная страница OpenAI-compatible provider в Kilo перечисляет несколько типичных проблем. Для LeanToken они разбираются так же.

### `Invalid API Key`

- проверьте, что вы вставили актуальный LeanToken key без лишних пробелов
- убедитесь, что используете именно тот ключ, который выпустили в `API Keys`

### `Model Not Found`

- проверьте корректность model ID
- убедитесь, что нужная модель действительно доступна через ваш LeanToken endpoint

### Ошибки подключения

- перепроверьте `Base URL`
- для стандартной схемы используйте `https://api.leantoken.tech/v1`
- если у вас нестандартный прокси, попробуйте полный endpoint URL только если он действительно требуется

### Ответы есть, но качество плохое

- проблема может быть не в LeanToken, а в выбранной модели
- попробуйте более сильную coding-oriented модель
- проверьте, не слишком ли мал context window для вашей задачи

### Контекст быстро переполняется

- сократите `Max Tokens`
- используйте более короткие сессии
- переключитесь на модель с большим контекстом
- удалите лишние сообщения или начните новую задачу после завершения предыдущего крупного блока работы

## Практический сценарий Kilo + LeanToken

Ниже простой рабочий поток, который обычно хорошо показывает, что все настроено правильно.

1. Подключите LeanToken как custom provider.
2. Выберите модель для coding-задач.
3. В `ask`-режиме попросите Kilo кратко описать структуру текущего проекта.
4. В `plan`-режиме попросите составить план изменения.
5. В `code`-режиме разрешите внести одну небольшую правку.
6. В `debug`-режиме разберите ошибку через `@terminal` или `@problems`.
7. После этого оцените, хватает ли вам качества, скорости и стоимости именно на вашем model ID.

## Что проверить после настройки

- Kilo Code отвечает без ошибки аутентификации
- выбранная модель действительно доступна через ваш LeanToken key
- ответы приходят через подключенный custom/OpenAI-compatible provider, а не через встроенный провайдер Kilo
- ключ не попал в `kilo.jsonc`, export JSON, публичные файлы или заметки команды
- выбранный agent и выбранная модель соответствуют типу задачи
- `@problems`, `@terminal` и обычный file/context flow работают предсказуемо

## Итог

Kilo Code хорошо подходит для BYOK через LeanToken, потому что в официальной документации у него есть прямой и понятный путь через `OpenAI Compatible` provider. Для базового старта достаточно четырех вещей: `Base URL`, `API key`, `Model` и аккуратных разрешений на действия агента.

Если делать практический вывод, то лучший путь такой:

1. сначала стабильно подключить LeanToken как custom provider
2. затем подобрать модель под ваш coding workflow
3. после этого уже настраивать agents, auto-approve и более сложные сценарии автоматизации
